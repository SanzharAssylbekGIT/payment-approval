## 1. Смета — первоклассная сущность системы

«Внутренняя смета» (полная калькуляция проекта) перестаёт быть внешним документом и
становится сущностью в БД. Она — источник правды для:
- разнесения поступлений (НДС / себестоимость / маржа);
- плана себестоимости по получателям;
- наполнения депозитов/резервов;
- план-факта по проекту.

**Состав сметы:** цена клиенту (с НДС / без НДС), сумма НДС, плановая себестоимость
с разбивкой по получателям/категориям, маржа, доля в депозит продакшна (для Influence).

**Владелец/момент ввода:** смета вносится в систему **до** прихода денег (в момент,
когда проект продан/согласован). Кто именно заносит (коммерческий блок сам или
бухгалтерия со слов коммерции) — **открытый вопрос (см. §11)**.

### 1.1. Смета изменяема и версионируется
Отклонения по сумме поступления исключены (клиент всегда платит ровно по смете).
Единственные источники изменений:
- менеджер прислал **не ту** смету (исправление);
- проект **сократили** в моменте (урезание себестоимости/цены).

Поэтому смета:
- **версионируется** — история изменений: кто, когда, с какой суммы на какую, **причина**;
- при изменении **пересчитывает леджер** — уже полученные поступления пере-разносятся
  пропорционально новой смете, плановые суммы получателей обновляются;
- **подсвечивает конфликт** — если уже выплачено больше, чем себестоимость в новой
  (урезанной) смете → флаг «перевыплата относительно сметы», разбирает бухгалтер/CFO.

## 2. Денежный контур (банк без API)

Прямого API для инициации **исходящих** платежей у Kaspi Business нет (их API — это
приём денег: Kaspi Pay / QR / эквайринг). Поэтому деньги всегда проводит человек,
а система готовит файлы. 1С — **обязательный** учётный контур (бухгалтерия всё равно
заносит платежи в 1С), поэтому интегрируемся через неё, без двойного ввода.

### Граница ответственности
```
НАША СИСТЕМА                          1С                       KASPI
согласование заявок            официальный учёт          проведение денег
управленческий учёт            бух. проводки             выписка
проектные балансы 7366
─────────────┬───────────────────────┬────────────────────────┬─────
реестр на оплату  ──экспорт файла──►  загрузка платежей  ──►   оплата
                                                                  │
статус «оплачено» ◄──разбор выписки── выписка (1с/excel) ◄───────┘
проектные балансы ◄──разнос по смете──┘
```

- **Наружу:** реестр на оплату отдаёт файл, бухгалтер **загружает** его в 1С (не
  перенабирает). 1С формирует платёжки в Kaspi.
- **Внутрь:** выписка из Kaspi → импорт в систему → автоматически закрывает «оплачено»
  и распознаёт поступления.

### Управленческий учёт — только у нас
Проекты 7366, депозиты/резервы, маржа, дашборды Клиент→Проект→Получатель ведутся
**только в нашей системе**, с супер-ограниченным доступом (CFO + бухгалтерия). 1С это
в нужном разрезе не показывает. Параллельного управленческого учёта в 1С нет.

## 3. Статус «оплачено» — по факту списания из выписки

«Оплачено» проставляется **только** когда платёж появился в банковской выписке
(фактическое списание), не в момент отправки в банк. Источник — импорт выписки.
Юридически источник правды остаётся бухгалтер (подтверждает спорные сопоставления),
но 95% работы система готовит автоматически из выписки.

**Форматы выписки:** приоритет — **1С (формат обмена)** как основной (структурирован,
парсится надёжно). Excel — через настраиваемый маппинг колонок. PDF — ручной фоллбэк
без автоматики.

## 4. Поступления и разнесение

### Обычный поток
```
банк → бухгалтер видит деньги (из выписки) → привязывает к проекту →
система предлагает разнос по смете (НДС→3098, себестоимость→7366, маржа→6890) →
бухгалтер подтверждает или правит
```

### Частичная оплата — пропорционально
Клиент платит частями. Каждое поступление разносится **пропорционально своей доле**
от суммы сметы: пришло 50% → НДС, себестоимость и маржа делятся 50/50. Баланс проекта
накапливается по поступлениям.

### Арифметика (зафиксировать в коде)
- Все суммы — целые в тиынах (BigInt), не float.
- **Округление:** остаток от деления (1 тиын на нечётных суммах) всегда сыпется в одну
  корзину (маржа на 6890), чтобы сумма частей **ровно** равнялась поступлению.
- Отклонений суммы поступления от сметы **нет** (см. §1.1) — ветку «хвост расхождения»
  не закладываем.

## 5. Поступление без сметы = SLA-нарушение (влияет на бонусы)

«Деньги пришли, сметы нет» — не технический статус, а измеряемое нарушение:
- фиксируется **ответственный менеджер**, **дата поступления**, **сколько дней висит**;
- система пингует менеджера, эскалирует при росте просрочки;
- копится **отчёт «просрочки по сметам»** (кто / проект / дней) → идёт в расчёт бонусов.

Деньги при этом лежат на 6890 как «не разнесено / ждёт сметы» и не блокируют работу.
Как определяется ответственный (автоматически по клиенту/проекту или назначает
бухгалтер) — **открытый вопрос (см. §11)**.

## 6. Получатель: план (смета) ↔ факт (заявка)

Получатель существует в двух связанных ролях:
- **План** — строка сметы: «блогер Иван — 100 000 — план»;
- **Факт** — заявка на выплату ссылается на плановую строку, после оплаты → факт.

```
Смета проекта
   └─ плановый получатель: Иван — 100 000 (план)
            ▲ заявка ссылается на плановую строку
   Заявка «оплатить Ивану 100 000» → согласование → оплачено (факт)
            ▼
   Баланс проекта 7366 = поступило от клиента − фактически выплачено
```

Платежи получателям **могут идти до** поступления от клиента → баланс проекта на старте
часто «−». План-факт ведётся внутри проекта по каждому получателю (виден перерасход).

### Запуск проекта (алгоритм)
1. Завели проект → внесли структуру себестоимости из сметы (плановые получатели).
2. Добавили платежи, прошедшие **до** оплаты клиента.
3. Сформировался леджер: баланс = поступления − выплаты.

## 7. Проект / Депозит / Резерв — один балансовый механизм

Проект на 7366, депозит продакшна (Influence) и резерв коммерческого продакшна
(Video/Photo) — **варианты одной балансовой сущности** («леджер с иерархией и балансом
приток/отток/остаток»). Отличаются только привязкой к счёту и тем, схлопывается ли
остаток в маржу.

**Депозит устроен как мини-7366:** внутри него — множество проектов (проект №1, №2 …),
заявки/поступления/остатки ведутся по **каждому уникальному проекту** депозита, со своим
балансом и детализацией. Не одна общая «копилка».

## 8. Что из этого следует для модели данных (превью)

- `Estimate` (смета) + `EstimateVersion` + `EstimateLine` (плановые получатели/категории).
- `Ledger` / `LedgerEntry` — единый механизм для проектов, депозитов, резервов
  (поле «тип» + привязка к счёту).
- `Incoming` (поступление) со статусом «не разнесено / разнесено» и привязкой к менеджеру.
- `Allocation` — разнос поступления по смете (пропорциональный, в тиынах).
- `BankStatementImport` + строки выписки + правила сопоставления с заявками.
- `PayoutFileExport` — файл реестра наружу (формат обмена 1С).
- Версионирование сметы и весь разнос — под аудит (`AuditLog`).

## 12. Заявки v2 — форма, срочность, доработка (решено 2026-07-02)

Уточнения по блоку «Мои заявки» (обсуждение с CFO). Дополняют/переопределяют §7–§12 CLAUDE.md.

### 12.1. Срочность вместо приоритета
Прежний «приоритет-критичность» (CRITICAL/RELATIONSHIP/FLEXIBLE) **заменён** на **срочность**
(`Urgency`), которую выбирает заявитель:
- **Срочно** — 1 рабочий день;
- **Средней срочности** — 2–3 рабочих дня;
- **Не срочно** — 5 рабочих дней.

Желаемая дата оплаты **не может быть раньше** минимума по срочности (в рабочих днях от подачи;
прошлые даты исключены). Казначейство сортирует реестр при дефиците по срочности + дате
(`URGENCY_RANK`). Чтобы налоги/ЗП всё равно всплывали наверх, у видов расхода задаётся
**срочность по умолчанию** (Зарплата → «Срочно»). Компромисс принят осознанно: явной отметки
«критично/юридически» больше нет.

### 12.2. Черновик / отправка / доработка
- Форма создания: две кнопки — **«Отправить»** (сразу на согласование) и **«Сохранить черновик»**.
- При уходе со страницы с несохранёнными данными — модалка «Сохранить черновик / Удалить»
  (+ нативное предупреждение браузера на закрытие вкладки).
- Черновик и заявку **«На доработке»** автор может редактировать (`/requests/[id]/edit`).
- Запрос уточнения согласующим возвращает заявку автору в статус **CLARIFICATION**
  (в UI — «На доработке»); при повторной отправке маршрут проходится заново с первой ступени.
- В списке «Мои заявки»: фильтр по статусу, колонки дата + срочность, подсветка «требует действия»
  (черновики и заявки на доработке).

### 12.3. Форма выплаты блогеру (вид расхода «Гонорары блогеров»)
Особая форма вместо общей:
- **Форматы работ** (мультивыбор): сторис, серия сторис, видеопост, фотопост, тикток, ютуб, прочее.
- **Сумма по договору** + **% от оплаты** (0–100). Сумма к оплате = договор × %
  (округление до тиына). % — доля от общей суммы по договору; выплаты фиксируются по проекту/получателю.
- **Предоплата / постоплата** и флаг **«услуга оказана»** — независимые поля. Если услуга оказана,
  при отправке **обязателен подписанный акт выполненных работ** (АВР).
- Убраны «строка сметы (план)» и «назначение платежа» (назначение формируется из форматов работ).

### 12.4. Вложения по категориям
`Attachment.kind`: договор / счёт / акт выполненных работ / сертификат резидентства / прочее.

### 12.5. Проекты
В форме заявки проект только **выбирается**, список фильтруется по услуге вида расхода
(Event → только Event-проекты). Создаёт проекты ответственный на экране
`/accounting/projects`: услуга определяет леджер (спецпроект → 0175, остальное → 7366),
департамент владельца определяет видимость проекта в форме (§10).
> Обновление (§13): право создавать проекты расширяется на **аккаунт-менеджеров**.

## 13. Целевая картина системы и связка с CRM (зафиксировано 2026-07-02)

Видение CFO — система как **финансовый контур компании**:
1. **Учёт проектов** — аккаунт-менеджеры заносят проекты (и сметы) сами.
2. **Все платежи компании** (проектные и офисные) проходят через систему.
3. Система — **внутренний источник правды по финансам**: все транзакции в ней.
4. **Леджеры проектов** (балансы Клиент → Проект → Получатель).
5. **План-факт по ВСЕМ расходам** (проектные — через сметы, офисные — через бюджеты).
6. **Статусы оплат** (оплачено/не оплачено) видны по проекту и по бюджету.
7. **Фиксация всех таймингов**: дата заявки, каждого согласования, оплаты — данные уже
   пишутся (createdAt / RequestApproval.decidedAt / Transaction.occurredAt / AuditLog),
   нужно сделать их видимыми (SLA-отчёт по срокам согласования).

**CRM — отдельная будущая система** (этапы сделок, воронка менеджеров). НЕ строим CRM-функции
внутрь финансовой системы. Для будущей связки закладываем:
- Проект/Клиент — связующие сущности со стабильными id (уже так);
- интеграция через API (route handlers) с идемпотентными операциями:
  «сделка выиграна в CRM → создать проект + смету здесь»;
- сервисный слой уже отделён от транспорта — API добавляется без переписывания логики.

### 14.3. Экономика сделки при создании
Сразу заносится: **сумма сделки** (с НДС), НДС (авто **16/116** — ставка РК с
2026, обновлено 2026-07-04; единственная точка изменения — `vatFromGross`),
сумма без НДС,
**себестоимость = продакшн-резервы + выплаты блогерам**.
**Обновлено 2026-07-04:** продакшн-резерв считается **по каждому блогеру и
каждой опции отдельно** (поле в каждой строке таблицы, EstimateLine.reserveAmount);
Σ резервов строк пишется в depositAmount версии (база будущего депозита).
Единого поля «резерв на сделку» больше нет.

### 14.4. База цен блогеров и скидки
Глобальный справочник **Blogger + BloggerPrice** (цена за формат). В таблице
сделки блогер выбирается из базы; прайс = Σ цен по выбранным форматам;
**скидка = прайс − заполненный гонорар** (фиксируется в строке сметы baseFee).
Форматы: сторис, серия сторис, **рилс**, фотопост, тикток, прочее (текстом).
Блогера не из базы можно ввести вручную (без прайса/скидки).

- Кнопка называется **«Создать проект»** и открывает **модальное окно**
  (не разворачивающийся блок).
- **Номер проекта** присваивает система: сквозная нумерация по компании
  (max+1), независимо от того, кто занёс проект и какая услуга
  (Project.number, уникален в entity). В форме номер показывается заранее.
- **Клиент — только из справочника**: выпадающий список с поиском, свободный
  текст не принимается. Добавление нового клиента — отдельное окно
  («+ Добавить клиента» в списке); дубликаты по имени не создаются.
- Пока форма открыта **только в разделе «Блогеры»** (Influence) — остальные
  разделы подключим следующими итерациями.
- При смене блогера в строке **опция остаётся**: если она есть в прайсе нового
  блогера — гонорар подставляется по его цене; своя опция переносится текстом.
- Живой расчёт: НДС · Без НДС · Себестоимость · **Продакшн** (Σ резервов) ·
  Маржа · **Маржа, %** (от суммы без НДС).

### 14.6. Карточка клиента (зафиксировано 2026-07-04)
Окно «Новый клиент» — полноценная карточка: **название** (как видно в системе),
**юридическое название**, **форма компании** (ИП/ТОО/АО/ЧК), **местная или
иностранная**, банковские реквизиты: **БИН** (12 цифр, у иностранных может
отсутствовать), **номер счёта** (казахстанский IBAN валидируется: KZ + 18),
**банк** (дропдаун со всеми БВУ РК + «Другой банк» вручную), **КБЕ — считает
система**: 1-я цифра 1/2 (резидент/нерезидент), 2-я — 9 для ИП, 7 для юрлиц
(17/19/27/29). Дубликаты по названию не создаются (просим выбрать из списка).
Справочник: `Client` (legalName, companyForm, isForeign, bin, bankAccount,
bankName, kbe), банки/формы/КБЕ — `src/lib/clients/constants.ts`.

## 15. Заявки на гонорары блогеров: опции из проекта и четверги (зафиксировано 2026-07-04)

### 15.1. Формат работ = утверждённая опция из сметы проекта
В заявке на гонорар блогера «формат работ» — **выпадающий список утверждённых
опций** из текущей сметы выбранного проекта (строки «блогер × опция» сделки).
Выбор опции автоматически подставляет блогера-получателя и (если поле пустое)
сумму по договору = гонорар из сметы. Назначение платежа берёт текст опции:
«Гонорар блогеру: {опция}». Заявка хранит ссылку на строку сметы
(estimateLineId). Чекбоксы форматов остаются только фолбэком для старых
проектов без строк сметы.

### 15.2. Выплаты блогерам — по четвергам + опция «Срочно»
Плановые выплаты блогерам проводятся **раз в неделю по четвергам**. В заявке
блогера срочность имеет два варианта:
- **«Плановая — ближайший четверг»** (по умолчанию): желаемая дата
  автоматически ставится на ближайший четверг; выбрать можно только четверг
  (не-четверг «примагничивается» к следующему четвергу, сервер отклоняет
  не-четверг);
- **«Срочно — 1 рабочий день»**: любой день, не раньше +1 раб. день.
Если сегодня четверг — «ближайший» значит следующий (реестр текущего уже
собирается). Правило живёт в validateDesiredDate (сервер) + RequestForm (UI).

### 15.3. Ставка НДС
НДС Казахстана в цене: **16/116** (ставка 2026). Точка изменения —
`vatFromGross` (src/lib/estimates/service.ts) и живые расчёты форм.

### 15.4. Видимость проектов: руководитель блока (зафиксировано 2026-07-04)
Правило §10 расширено и вынесено в единую точку (`src/lib/projects/scope.ts`,
projectScopeFilter — обязателен для ЛЮБОГО запроса проектов): проект видят
владелец-продажник, прикреплённый проджект, департамент владельца, а
**руководитель блока (роль APPROVER) — ещё и проекты, которые ведут проджекты
его департамента** (Рахима видит все проекты, где проджект из блогерского
блока). Рядовые проджекты по-прежнему видят только свои проекты.

## 16. Проекты «Продакшн» (Video/Photo): смета из Excel (зафиксировано 2026-07-04)

Кнопка «Создать проект» открыта и в разделе «Продакшн». Форма та же, что у
блогеров (сквозной номер, клиент из справочника, проджект, даты, сумма сделки,
НДС 16/116 авто), но блок себестоимости другой:
- вместо таблицы блогеров — **«Себестоимость проекта»**: продюсер загружает
  **Excel-файл (.xlsx)** со сметой, система читает его и выводит строки
  в редактируемую таблицу прямо в окне (обязательная ручная проверка);
- парсер-эвристика (src/lib/estimates/excel.ts): в каждой строке листа первый
  текст — название, последнее положительное число — сумма (формулы
  «кол-во × цена» берутся по результату); строки «Итого/Всего/Total»
  пропускаются; берётся первый лист, где нашлись строки;
- сам файл НЕ сохраняется — источник правды: смета в системе после правок;
- строки можно добавлять/править/удалять вручную (и вовсе без файла);
- строки становятся получателями-подрядчиками проекта (план/факт на карточке),
  продакшн-резерва per-строка у продакшна нет (остаток себестоимости → резерв
  коммерческого продакшна при закрытии — по ТЗ §5, в бэклоге депозитов).
Раздел «Ивенты» — следующая итерация.

### 16.1. Нумерация проектов — по направлениям (зафиксировано 2026-07-04)
Вместо сквозной нумерации по компании — **своя последовательность у каждого
направления** с префиксом: **IM-#** блогеры (Influence), **VP-#** продакшн,
**EM-#** ивенты, **SP-#** спецпроекты. Номер присваивает система (max+1
внутри направления), уникальность (entityId, serviceType, number).
Префиксы/формат — src/lib/projects/code.ts (projectCode). Существующие
проекты перенумерованы внутри направлений с сохранением порядка.

### 16.2. Заявки продакшна: несколько позиций сметы разом (зафиксировано 2026-07-04)
В заявке по проектному виду расхода (кроме гонораров блогеров) позиции сметы
выбираются **мультивыбором** (чекбоксы) — один платёж может закрывать
несколько статей (например, камера + свет одному подрядчику). Сумма заявки и
назначение платежа подставляются автоматически из выбранных позиций
(редактируемы); одна позиция → получатель подставляется из строки. Связь
хранится в таблице payment_request_lines (заявка ↔ строки сметы);
одиночное поле estimateLineId заполняется, когда позиция ровно одна.
Сервер отклоняет позиции не из сметы выбранного проекта.

### 16.4. Дробление платежей продакшна (зафиксировано 2026-07-04)
В проектных заявках (кроме гонораров блогеров) платёж дробится: поле
**«% от суммы позиций»** (1–100, по умолчанию 100) + предоплата/постоплата.
Сумма заявки = Σ выбранных позиций × % (авто, half-up до тиына, можно
поправить руками). Контроль: **суммарный % по КАЖДОЙ позиции сметы** по всем
«живым» заявкам ≤ 100 (50% предоплата + 50% постоплата проходят, третья
заявка отклоняется с указанием позиции) — assertLinePercentLimit.
