## 1. Учёт / Система Б (самое важное)

- [ ] **Смета: UI создания/редактирования** — сметы сейчас появляются только через
      сид/скрипты. Экран для **аккаунт-менеджеров** (решено, DECISIONS §13):
      заносят проект + смету сами; расширить право createProject на аккаунтов
      (сейчас только бухгалтерия/CFO) + пересчёт разнесений при новой версии
      (DECISIONS §1.1). **← СТАРТ ЗДЕСЬ**
- [ ] **Депозиты и резервы не работают** (§4–5 ТЗ): `EstimateVersion.depositAmount`
      игнорируется при разнесении, `DEPOSIT_FUNDING`/`RESERVE_FUNDING` нигде не
      создаются, дашборд «Депозиты и резервы» всегда пуст. Нужно: при разнесении
      Influence-поступления отщеплять продакшн-бюджет в депозит-леджер; для
      Video/Photo — остаток себестоимости в резерв при закрытии проекта.
      (Зависит от смет — идёт вторым.)
- [ ] **Тайминги/SLA** (DECISIONS §13.7): данные уже пишутся (createdAt,
      RequestApproval.decidedAt, Transaction.occurredAt, AuditLog) — сделать
      видимыми: сроки на карточке заявки, отчёт «сколько висело на согласовании».
- [ ] Частичное разнесение (`PARTIALLY_ALLOCATED`) — статус есть, логики нет.

## 2. Импорт выписки

- [ ] **Дедупликация**: повторная загрузка того же/пересекающегося файла дублирует
      строки; каждый дубль CREDIT можно превратить во второй Incoming. Нужен
      fingerprint строки (дата+сумма+докномер) и/или хэш файла.
- [ ] **Миграция xlsx → exceljs**: пакет `xlsx` имеет high-уязвимости без фикса
      (prototype pollution, ReDoS); `exceljs` уже в devDependencies.
- [ ] Сверка «строка выписки ↔ заявка»: авто-матч по сумме+дате+получателю.

## 3. Согласование

- [ ] **Параллельные согласующие на одной ступени**: схема разрешает
      (`@@unique([routeId, order, approverId])`), движок берёт первого
      (`currentStep` → `.find()`). Либо поддержать, либо запретить в админке.
- [ ] Уведомления (email/Telegram) о заявках «на мне» — сейчас только pull.
- [ ] Делегирование/замещение согласующего на отпуск.

## 4. Заявки

## Принятые допущения (не баги)

- Оплата APPROVED-заявки возможна только через реестр (UI); банковский импорт —
  отдельный путь, там факт списания первичен.
- Лимит «суммарный % по блогеру ≤ 100» считает и черновики (заявленные доли).
- Сид безопасен на живой БД: ступени маршрутов синхронизируются без удаления
  ступеней с историей решений.
