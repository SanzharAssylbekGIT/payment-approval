## 1. Назначение продукта

Внутренняя веб-система Brave Talents, заменяющая согласование платежей через WhatsApp/почту и ручную сборку финансовых отчётов. Состоит из **двух связанных подсистем**:

- **Система А — Заявки и согласование платежей.** Сотрудник создаёт заявку → маршрутизация согласующему(им) по виду расхода → одобрение → платёжный календарь → реестр на оплату → проведение в банке (вручную) → отметка «оплачено».
- **Система Б — Управленческий учёт.** Логика счетов компании и проектный учёт оборотного счёта. Автоматически формирует отчётность, которую сейчас бухгалтерия собирает вручную.

Связь систем: одобренный и оплаченный платёж автоматически отражается в нужном счёте/проекте/депозите Системы Б.

## 2. Технический стек

- **Frontend + Backend:** Next.js (App Router, React, TypeScript)
- **БД:** PostgreSQL
- **ORM:** Prisma
- **Контейнеризация:** Docker + docker-compose (для разработки и для передачи на деплой айтишнику на офисный сервер)
- **Авторизация:** на MVP — логин/пароль (хэш bcrypt/argon2, сессии). Архитектурно заложить переключение на SSO (Google Workspace / Microsoft Entra) одним модулем — провайдер авторизации абстрагировать, чтобы добавить OAuth позже без переписывания.
- **Масштаб:** 70+ пользователей.
- **Мультиарендность:** старт — только Brave Talents, но модель данных закладывает `entity_id` (юрлицо/компания) во все ключевые таблицы, чтобы позже подключить другие entities Brave Group без миграции схемы.

### Принципы разработки
- Начинать с модели данных (Prisma schema) и миграций, затем API, затем UI.
- Все денежные суммы хранить в минимальных единицах (тиын) целым числом (BigInt) либо Decimal — НЕ float. Округление — по бухгалтерским правилам.
- Каждое изменение статуса заявки/платежа писать в журнал аудита (кто, когда, что).
- Конфиденциальность — на уровне запросов к БД (фильтрация по роли и принадлежности к блоку), не только в UI.

## 3. План счетов и финансовая логика

### Счета
- **6890** — основной счёт. Сюда поступают деньги от клиентов. Здесь остаётся маржа. Отсюда платятся офисные расходы и зарплата.
- **7366** — оборотный счёт себестоимости проектов. Из него платятся блогеры/подрядчики. Учёт по иерархии Клиент → Проект → Получатель.
- **3098** — счёт НДС.
- **0175** — счёт спецпроектов (доходная + расходная части + маржа на одном счёте, только НДС выводится на 3098).

### Логика разнесения поступления (для обычных услуг, не спецпроектов)
При поступлении средств от клиента на 6890 сумма дробится на три части:
1. **НДС** → переводится на 3098
2. **Себестоимость** → переводится на 7366 (привязка к конкретному проекту)
3. **Маржа** → остаётся на 6890

### Логика спецпроектов
Всё (доход + себестоимость + маржа) учитывается на едином счёте **0175**. Только НДС выводится на 3098. Отдельных переводов себестоимости нет.

## 5. Депозиты и резервы («оборотные копилки»)

Это сущности, где **остаток себестоимости НЕ схлопывается в маржу**, а сохраняется под будущие нужды (например, зарплаты продакшн-команды). Ведутся по той же балансовой логике, что и проекты на 7366: видно, **где потрачено, но не получено**, и **где получено, но не потрачено**.

- **Депозит продакшна (Influence)** — формируется из продакшн-бюджета. Пример: заложено 200 000, потрачено 100 000, остаток 100 000 доступен на другие нужды. Внутри депозита — приток/отток с балансом.
- **Резерв коммерческого продакшна (Video/Photo)** — формируется из остатков себестоимости коммерческих съёмок. Та же логика притока/оттока с балансом.

Оба должны показывать баланс и детализацию операций (приток / отток / остаток), аналогично проектному учёту.

## 6. Проектный учёт на 7366 (ядро Системы Б)

### Иерархия
**Клиент → Проект → Получатель**

- **Клиент** — например, «Яндекс Поиск».
- **Проект** — у клиента может быть несколько: «Наурыз», «Техноблогеры», «8 марта».
- **Получатель** — блогер/подрядчик внутри проекта, со статусом оплаты.

### Балансы (схлопываются на каждом уровне)
- **По проекту:** баланс может быть «+» (клиент оплатил, получателям ещё не выплачено) или «−» (получателям выплачено, клиент ещё не оплатил).
- **По клиенту:** сумма балансов всех его проектов.
- **Общий баланс 7366:** сумма по всем клиентам.

### Детализация проекта (экран «проваливания»)
При открытии проекта видно:
- Список получателей: Имя — Сумма — Статус (оплачен / не оплачен)
- Баланс по проекту
- Счётчики: сколько получателей оплачено / сколько к оплате

## 8. Казначейство (treasury)

### Платёжный календарь
Все одобренные заявки попадают в календарь с желаемой датой оплаты. На горизонте недели/месяца видно: ожидаемые поступления (от клиентов) vs ожидаемые выплаты (заявки). Кассовый разрыв виден заранее.

### Реестр на оплату
Раз в день/неделю формируется реестр одобренных заявок к оплате. Отсортирован по приоритету. Показывает остатки на счетах. На основе реестра принимается решение, что проводить сегодня.

### Приоритизация при дефиците
Приоритеты (по убыванию критичности):
1. **Критичные/юридические:** налоги, зарплата, обязательства с пенями/штрафами.
2. **Удерживающие отношения:** блогеры/подрядчики действующих проектов (срыв = срыв проекта).
3. **Гибкие:** подписки, подарки — можно перенести.

Дополнительно: выплаты по проекту 7366 с отрицательным балансом (клиент ещё не заплатил) можно притормаживать — система это подсвечивает.

### Решение при дефиците — коллегиальное
Система сортирует реестр по приоритету и предлагает план. Финальное решение «проводим сегодня вот это» принимают совместно: **Санжар (CFO) + Жадыра (гл. бух) + Айнур (опер. директор)**. Заявки ниже доступного лимита переносятся на следующий период.

## 9. Бюджет и дашборды

### Бюджет 6890 (план-факт)
По счёту 6890 (кроме дивидендов Алмаса) есть отдельный плановый бюджет по статьям. Нужно:
- Ввод плана по статьям расходов.
- Автоматический факт по фактически проведённым платежам.
- План-факт-отклонение, % освоения.
- Проваливание в детали по каждой статье (список конкретных платежей).

### Дашборды
- **Освоение бюджета 6890** — план/факт/остаток по статьям, прогресс-бары.
- **Проектные балансы 7366** — Клиент → Проект → Получатель, балансы на каждом уровне.
- **Депозиты и резервы** — балансы и движения.
- **Платёжный календарь** — поступления vs выплаты.
- **Ежемесячный отчёт** (автоматизация текущего ручного отчёта бухгалтерии): Поступления (оборот с НДС, оборот без НДС, себестоимость, маржа), Расходы себестоимости, Расходы офисные, Спецпроекты.

## 10. Роли и права

| Роль | Кто | Права |
|---|---|---|
| **Заявитель** | сотрудники отделов | Создаёт заявки своего вида расхода. Видит только свои заявки и их статусы. |
| **Согласующий** | руководители блоков (Рахима, Айнур, Ажар, Кристиана, Каламкас) | Видит и согласует только заявки своего блока. Видит остатки/проекты только своего блока. |
| **Казначей / CFO** | Санжар | Видит ВСЁ. Ведёт реестр, платёжный календарь, участвует в коллегии при дефиците. |
| **Бухгалтер** | Жадыра, Сымбат, Эльнура | Заносят платежи в банк (вне системы), ставят статус «оплачено» по факту. Ведут зарплату. Жадыра — гл. бух, согласует зарплату и дивиденды, входит в коллегию. |
| **Админ** | назначается | Настраивает маршруты, виды расходов, добавляет/удаляет пользователей, права. |

### Двойные роли
- **Айнур Абдували** — согласующий (несколько видов) + член казначейской коллегии.
- **Жадыра Касымбек** — гл. бух (согласует зарплату/дивиденды) + член казначейской коллегии.

### Кто ставит «оплачено»
**Бухгалтер**, по факту оплаты (после того как CFO провёл платёж в банке). Это единственный источник правды о фактической оплате.

### Конфиденциальность (КРИТИЧНО)
- Каждый видит **только свои остатки/проекты/заявки**.
- Рамзат — только свои съёмки; Рахима — только блогерский блок; и т.д.
- Полную картину (все проекты, все балансы, казначейство, отчёты) видят **только CFO и бухгалтерия**.
- Фильтрация прав — на уровне запросов к БД, не только в интерфейсе.

## 11. Рабочие экраны (UI)

1. **Панель подачи заявки** (Заявитель): форма — вид расхода, клиент/проект (если проектный), получатель, сумма, назначение платежа, прикреплённый счёт/договор (файл), желаемая дата оплаты, приоритет, комментарий. После отправки — статус заявки.
2. **Панель согласования** (Согласующий): очередь заявок «на мне», по каждой — одобрить / отклонить / запросить уточнение. Только свой блок.
3. **Реестр на оплату + платёжный календарь** (CFO/коллегия): одобренные заявки, сортировка по приоритету, остатки на счетах, отметка к проведению.
4. **Отметка оплаты** (Бухгалтер): список проведённых, простановка статуса «оплачено».
5. **Дашборды** (§9): бюджет, проектные балансы, депозиты/резервы, календарь, ежемесячный отчёт.
6. **Админка** (Админ): пользователи, роли, виды расходов, маршруты.

## 12. Статусы заявки (жизненный цикл)

`Черновик` → `На согласовании` (по ступеням маршрута) → `Одобрена` / `Отклонена` → (если одобрена) `В реестре на оплату` → `Оплачена`.

Каждый переход логируется (аудит): кто, когда, действие, комментарий.

## 14. Очередность разработки (для Claude Code)

1. **Каркас:** Next.js + TypeScript + Prisma + PostgreSQL в Docker; docker-compose; базовая структура папок.
2. **Модель данных:** Prisma schema со всеми сущностями (Entity, User, Role, Client, Project, Recipient, ExpenseType, ApprovalRoute, PaymentRequest, Account, Deposit/Reserve, Transaction, BudgetLine, AuditLog). Миграции.
3. **Авторизация:** логин/пароль, роли, middleware прав (абстракция под будущий SSO).
4. **Система А:** панель подачи заявки → маршрутизация → панель согласования → жизненный цикл статусов.
5. **Казначейство:** реестр на оплату, платёжный календарь, приоритизация, отметка «оплачено».
6. **Система Б:** проектный учёт 7366, депозиты/резервы, разнесение поступлений, спецпроекты 0175.
7. **Бюджет и дашборды:** план-факт 6890, дашборды, ежемесячный отчёт.
8. **Админка:** управление пользователями, маршрутами, видами расходов.

> На каждом этапе — заполнять тестовыми данными и проверять права/конфиденциальность.
> Конфиденциальность и аудит — не опциональны, закладываются с самого начала.
